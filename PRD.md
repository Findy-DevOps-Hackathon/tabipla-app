## 1. プロダクト概要

### 1.1 一言で
**「今ここから行ける、自分だけのワクワクする観光」を、AIエージェントが現地で自律的に提案してくれる観光ガイドアプリ。** 各スポットには、その土地ならではの蘊蓄をAIがあなた向けに語ってくれる。

### 1.2 解決する課題

**観光客（toC）の課題**
- 定番TOP10・映えスポットに飽き、自分に合うニッチな観光地が見つからない。
- マイナーな観光地は「実際に行って楽しいか」が事前に分からず不安。
- 旅行中に空き時間ができても、自分に合う場所を即座に提示してくれる手段がない。

**自治体（toB/toG）の課題**
- 近隣の有名観光地にブランドで負け、自地域の名産・歴史をアピールしきれない。
- 「映え」では勝てないが、その土地ならではの魅力（蘊蓄・ストーリー）を伝える場がない。

### 1.3 提供価値
- ユーザーには「**自分で見つけた感**」と「**人に語りたくなる蘊蓄**」を伴う観光体験を、**AIエージェントとの対話・自律提案**を通じて。
- 自治体には「**蘊蓄ネタを注入すればAIが魅力的に語ってくれる掲載面**」を。

## 2. AIエージェントのコンセプト（ハッカソンの中核）

### 2.1 旅サキ・エージェント（現地コンシェルジュ）
旅行中のユーザーに伴走し、**「今ここから行ける、自分に合うニッチ観光地」を自律的に提案する**AIエージェント。

**エージェントへの入力（コンテキスト）**
- 現在地・滞在場所
- 残り時間・所要時間
- 予算・料金
- 移動手段（徒歩／車／公共交通）
- ユーザーの好み（オンボーディングで入力）

**エージェントの自律的な振る舞い（2段階で実現）**
1. **推薦＋理由提示型**（基礎）：条件に合うスポットをAIが選び、「**なぜあなたに合うのか**」を語って提示する。
2. **自律計画型**（発展）：複数スポットを自律的に探索・組み合わせ、空き時間に収まる**時間軸付きの小さな旅程**を一括提案する。

→ MVPでは(1)を確実に成立させ、余裕があれば(2)へ拡張する（優先度は §8 参照）。

### 2.2 蘊蓄AI（パーソナライズド・ストーリーテラー）
各スポットの「**人に語りたくなる蘊蓄**」をユーザー向けに語るAI。

- **重要原則：蘊蓄はゼロから生成しない。** 事実の素材（ネタ）は**管理側（自治体）が注入**する。
- AIの役割は、注入された蘊蓄ネタを、**そのユーザーの好み・文脈に合わせてパーソナライズし、語り口を生成する**こと。
- これにより、事実の正確さ（自治体由来）と、刺さる伝え方（AI由来）を両立する。

## 3. 対象ユーザー（ペルソナ）

### 3.1 観光客ペルソナ：佐藤 至恩（26歳・バックエンドエンジニア）
- 3ヶ月に1回旅行、年1回海外。国内35都道府県を制覇済み、リピートも多い。
- 普段はiPhoneで検索。Instagram・Google・ChatGPTで観光地を調べる。
- 酒蔵・神社・パン屋・カフェ・遺産・自然を好む。
- 求めるのは「ニッチでワクワクする観光地」「自分で見つけた感」「同行者にも語れる体験」。

### 3.2 自治体ペルソナ：武田 直人（35歳・市役所 商工観光課）
- 小諸市の観光振興担当。近隣（軽井沢等）に観光客を奪われている。
- ワイン・蕎麦・信州味噌・日本酒などの資源があるが、アピールしきれていない。
- 求めるのは「軽井沢とは違う自地域の魅力を伝えられる手段」。

## 4. スコープ

### 4.1 今回のMVPスコープ
- ✅ **旅サキ（toC・モバイルアプリ）** — AIエージェントによるシーンB（旅行中の空き時間）提案を主役に
- ✅ **蘊蓄AI** — 管理側が注入したネタをAIがパーソナライズして語る
- ✅ **旅サキ管理（toB/toG）** — ただし**最小化**（§6参照）
- ✅ クーポン／割引・現地認知拡散の仕組み

### 4.2 対応プラットフォーム
- **iOS優先**（ペルソナがiPhone利用のため）。AndroidはMVP後の拡張。
- 旅サキ管理は **Webサイト**（最小構成）。

### 4.3 今回スコープ外（将来拡張）
- 旅行計画の**メンバー共有**機能（MVPは保存まで）
- 動画・360度画像など高度な視覚表現
- Android対応
- リッチな管理サイト（多機能な入稿UI・分析ダッシュボード等）

## 5. 機能要件 — 旅サキ（toC アプリ）

### 5.1 アカウント・オンボーディング
- **アカウント登録必須。**
- 初回オンボーディングで**好み**を入力（例：酒蔵・神社・カフェ・自然・遺産 等）。
- 入力した好みがエージェントの推薦コンテキストになる。

### 5.2 AIエージェントによる現地即時提案【MVPの主役 / シーンB】
- ユーザーが現在地・残り時間・予算・移動手段を指定（または自動取得）すると、AIエージェントが条件に合うニッチ観光地を提案。
- 各提案には「**なぜあなたに合うのか**」の理由が添えられる。
- （発展）空き時間に収まる時間軸付きの小さな旅程を一括提案する。

**絞り込み条件（必須サポート）**

| 条件 | 内容 |
|------|------|
| 現在地・移動距離 | 現在地／滞在場所からの距離・アクセス |
| 残り時間・所要時間 | 空き時間に収まるか |
| 予算・料金 | 入場料・体験料などのコスト |
| 移動手段 | 徒歩／車／公共交通。到達可能範囲の算出に反映 |

### 5.3 蘊蓄体験（中核機能）
- 各スポットで、AIが**その土地ならではの蘊蓄**をユーザー向けに語る。
- ネタの出所は管理側（自治体）注入。AIはパーソナライズと語り口生成を担う。
- ユーザーが「**人に語りたくなる**」体験を得られることを差別化の核とする。

### 5.4 推薦のパーソナライズ根拠
- **好み入力ベース**（オンボーディング）を基礎とする。
- **行動・履歴ベース**（訪問済み地域、いいね／スキップ等からの学習）でも精度を上げる。
  - ※1ヶ月の開発制約上、行動学習は優先度Should（§8）。MVPでは好み入力ベースを確実に成立させる。

### 5.5 計画の保存（シーンA）
- 気に入った観光地を「行きたいリスト／お気に入り」として**保存**できる。
- ※メンバーへの共有はスコープ外（将来拡張）。

### 5.6 クーポン・現地認知
- スポット・店舗に紐づく**クーポン／特典**を提示・利用できる。
- 現地（ホテル・観光案内所・飲食店）でのアプリ紹介と連動し、口コミ・再訪・インストール拡散の導線とする。

## 6. 機能要件 — 旅サキ管理（toB/toG・最小構成）

> 1ヶ月の開発制約により、管理サイトは**最小化**する。toC体験の成立に必要な「データ注入」に役割を絞る。

### 6.1 役割
- 自治体が、推薦・蘊蓄・クーポンの**素材（ネタ）を注入する**ための最小限の入口。

### 6.2 注入できるコンテンツ
| 種別 | 内容 |
|------|------|
| 写真・テキスト説明 | スポットの基本情報（名称・位置・説明・画像） |
| 蘊蓄ネタ（素材） | その土地ならではの歴史・背景・事実。**AIが語り口を生成する元ネタ** |
| クーポン・特典 | スポット・店舗に紐づく割引・特典 |

- ※動画・360度画像はスコープ外。

### 6.3 公開フロー
- **自治体が直接公開**（運営審査なし）。スピードと運用負荷軽減を優先。

### 6.4 最小化の方針
- 凝った入稿UIは作らず、**最小限の入稿フォーム**、または**シードデータ／運用代行での投入**で成立させる。
- デモでは小諸市など特定地域のデータで世界観を見せる。

### 6.5 データ供給の全体像
- **自治体注入**（主ソース・蘊蓄ネタの源泉）。
- **外部データ**（既存観光DB・Web情報）でカバレッジ補完。品質ばらつきは前提。
  - ※外部データの本格活用は優先度Could（§8）。MVPは自治体注入を主軸に。

## 7. 事業・収益モデル

- **自治体課金を前提とする。** 自治体が掲載・ブランディングの対価を支払うモデルを基本とする。
- クーポンは観光客の体験向上と認知拡散（口コミ・インストール）の手段。
- ※MVP／ハッカソンでは課金実装そのものは対象外。あくまでモデルとして明記する。

## 8. MVP優先度（MoSCoW・1ヶ月で作り切るための絞り込み）

| 優先度 | 機能 | 理由 |
|--------|------|------|
| **Must** | AIエージェントによる現地即時提案（推薦＋理由提示） | ハッカソンの主役。これが無ければ成立しない |
| **Must** | 4条件での絞り込み（現在地・残り時間・予算・移動手段） | シーンBの価値の前提 |
| **Must** | 蘊蓄AI（管理側ネタ → AIがパーソナライズして語る） | 差別化の核 |
| **Must** | オンボーディング好み入力＋アカウント | 推薦のシード |
| **Must** | 管理側の最小注入（フォーム or シード） | toC体験のデータ源 |
| **Should** | クーポン提示・利用 | 認知拡散の導線。デモで効くが主役ではない |
| **Should** | 行動・履歴学習による精度向上 | 価値は高いが1ヶ月では重い。好み入力で代替可 |
| **Should** | 自律計画型（時間軸付き旅程の一括提案） | デモインパクト大。Must成立後に拡張 |
| **Could** | 外部データの本格活用 | 品質担保コスト高。デモは自治体データで十分 |
| **Won't（今回）** | 計画のメンバー共有／動画・360度／Android／リッチ管理UI | 将来拡張 |

## 9. コア体験フロー（シーンB・文章レベル）

1. 旅行中、予定が前倒し／休館などで**時間が余る**。
2. アプリを開き、AIエージェントに状況（残り時間・現在地など）が渡る。
3. AIが**条件に合うニッチ観光地を提案**し、「なぜあなたに合うか」を語る。
4. スポットを選ぶと、AIが**その土地ならではの蘊蓄**をあなた向けに語る。
5. ユーザーは「ここ良さそう！」と納得して向かう／お気に入りに保存する。
6. （発展）AIが空き時間に収まる**小さな旅程**を一括提案する。

## 10. 成功の定義（参考指標の方向性）

> 具体的な目標数値は別途設定。方向性のみ記載。

**ハッカソン観点**
- AIエージェントが「条件に応じて自律的に・的確に・魅力的に」提案できているか。
- 蘊蓄が「人に語りたくなる」品質でパーソナライズされているか。

**プロダクト観点（toC）**
- 提案された観光地への「行きたい／行った」転換、「自分で見つけた感」の充足。

**プロダクト観点（自治体）**
- 蘊蓄ネタの注入数、自地域への流入の手応え。
